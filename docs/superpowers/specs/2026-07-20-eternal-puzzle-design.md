# ペントミノ・パッキングパズル Webアプリ 設計書

- 作成日: 2026-07-20
- ステータス: ドラフト（レビュー待ち）

## 1. 概要

ポリオミノのパッキングパズルを Web で実装する。第一弾として
**ペントミノ 6×10**（12ピース・解 2,339 通り）を対象とする。

12 種のペントミノをすべて 6×10 の盤にすき間なく敷き詰めるパッキングパズル。
解が膨大にあり何度でも遊べる点が特徴。見つけた解を図鑑のように集めていく
**コレクション型**の体験を軸にする。

### スコープ

- **今回作るもの**: ペントミノ 6×10 の 1 種類のみ。
- **想定デバイス**: PC（マウス）中心。ドラッグ＆ドロップ ＋ キーボードで回転／裏返し。
- **やりこみ要素**: 発見した解を保存し「2,339 通り中 ○通り発見」と図鑑化。
- **将来拡張（今回は作らない）**: テトロミノ等の他ポリオミノ盤、ポリアモンド（三角形）系、
  スマホ最適化、タイムアタック。`puzzle/` にデータを足すだけで盤を増やせる構造にしておく。

### 非目標（YAGNI）

- バックエンド／アカウント／ランキング。
- ヒント・自動求解。
- 三角形グリッド（ポリアモンド）。

## 2. 技術スタック

- Vite + React + TypeScript。
- 状態管理は `useReducer` + Context。外部状態ライブラリは入れない。
- ドラッグは Pointer Events を薄い自作フックで実装（ネイティブ HTML5 DnD は使わない）。
- 永続化は localStorage のみ（バックエンドなし）。
- テストは Vitest。純ロジック（`puzzle/`）を重点的に。

## 3. モジュール構成

UI とパズルロジックを疎結合にする。`puzzle/` は React 非依存の純ロジックとして
単体テストで固める。

```

  state/
```

## 4. データ構造

### ピース（`pieces.ts`）

各ペントミノを「相対座標の集合」で定義。基準の向きを 1 つ持ち、実行時に回転・裏返しで
最大 8 方向を生成し、重複を除いてユニークな向きだけ保持する。

```ts
type Cell = { r: number; c: number };
type Orientation = Cell[];            // 正規化済み（最小の r,c が 0）の 5 セル

type PieceId = 'F'|'I'|'L'|'N'|'P'|'T'|'U'|'V'|'W'|'X'|'Y'|'Z';

type PieceDef = {
  id: PieceId;
  color: string;                      // 図鑑・盤で識別する色
  base: Cell[];                       // 基準形
};

type Piece = {
  id: PieceId;
  color: string;
  orientations: Orientation[];        // ユニークな向き（X は 1, I は 2, F は 8 …）
};
```

- 回転: `(r,c) -> (c, -r)`
- 裏返し: `(r,c) -> (r, -c)`
- 変換後は `r,c >= 0` になるよう正規化。8 通り生成し集合として重複排除。

### 盤と配置状態（`board.ts` / state）

```ts
const ROWS = 6, COLS = 10;           // 60 セル = 12 ピース × 5

type Placement = {
  pieceId: PieceId;
  orientation: number;               // orientations の index
  anchor: Cell;                      // 盤上のアンカー位置
};

type GameState = {
  placements: Record<PieceId, Placement | null>; // null = トレイに未配置
};
```

- `anchor` は、正規化済みオリエンテーション（最小の r,c が 0）の原点 (0,0) を
  対応させる盤上のセル。ピースの実占有セルは `anchor + orientation の各セル` で求める。
- 盤の占有状況は `placements` から**導出**する（単一の真実の源）。
- 合法配置の条件: 全セルが盤内 && 既存の他ピースと重ならない。
- ドラッグ確定時にこの判定を通ったものだけ置ける。

### 完成判定

12 ピース全部が盤に配置され、60 セルが過不足なく埋まっている状態。合法配置しか
許さないので、実質「12 個すべて盤上」で完成。

### 解（`solution.ts`）

完成盤を「60 セルそれぞれをどのピースが覆うか」の 1 枚の盤として捉える。

```ts
type SolutionGrid = PieceId[];       // length 60, 行優先
```

## 5. 操作モデル

- **つかむ**: トレイまたは盤上のピースを pointerdown で持ち上げ、カーソルに追従。
- **回転／裏返し**: 選択中のピースに対し `R`＝90°回転、`F`＝裏返し（ボタンでも可）。
  ドラッグ中もプレビューが即更新。
- **スナップ＆プレビュー**: ドラッグ中は最寄りセルにスナップした半透明プレビューを表示し、
  合法なら緑・不可なら赤でハイライト。
- **置く**: 合法な位置で離すと確定。不可なら元の場所（トレイ or 直前の盤位置）に戻る。
- **外す**: 盤上のピースをトレイへドラッグ、またはダブルクリックで取り外し。
- 「選択中のピース」概念を持ち、クリックで選択 → キーボードで回転/裏返し。

## 6. 重複排除ロジック（図鑑の肝）

**問題**: 6×10 の盤は左右反転・上下反転・180°回転すると別の見た目になるが、パズルとしては
同じ解とみなす（2,339 という数もこの数え方）。単純に見た目で保存すると 1 つの解を最大 4 回
重複カウントしてしまう。

**解法**: 完成盤 `SolutionGrid` に盤の対称変換 4 通り（恒等・180°回転・左右反転・上下反転
※正方形でないので 90°は無し）をすべて適用し、それぞれ文字列化。辞書順最小のものを
**正規形**として採用する。

```ts
function canonicalKey(grid: SolutionGrid): string {
  const t = [identity, rotate180, mirrorH, mirrorV].map(fn => serialize(fn(grid)));
  return t.sort()[0];          // 4 つのうち辞書順最小 = 一意なキー
}
```

- 各変換は 6×10 のインデックス張り替えだけで実装（回転は配列反転、左右反転は列反転、
  上下反転は行反転）。
- ピースのラベル（F は F）はセルと一緒に動くので、変換後も正しくラベル付き解として扱える。
- 図鑑は「発見済み正規形キーの Set」。**発見数 = Set のサイズ**、母数は 2,339。

## 7. 図鑑・永続化

### 図鑑（Collection）

- 「2,339 通り中 ○通り発見」を大きく表示。発見済みの解をサムネイル（縮小した盤の色配置）で一覧。
- 各サムネクリックで、その解を盤に再現表示（鑑賞用）。
- 発見時のみ日時を記録（`{ key, foundAt }`）。

### 永続化（`storage/collection.ts`）

- localStorage キー `eternal-puzzle:pentomino:collection` に発見済み `{key, foundAt}[]` を保存。
- 進行中の盤（`placements`）も `...:progress` に自動保存し、リロードで復帰。
- スキーマに `version` を持たせ、将来の盤追加でも壊れないように。

## 8. エラー処理・エッジケース

- 不正配置は確定させず視覚フィードバックで戻す（データ不整合を作らない）。
- localStorage 無効／容量超過時は try/catch でメモリ内動作にフォールバック（保存不可を軽く通知）。
- 完成時、正規形が既存キーなら「既に発見済み」、新規なら「新発見！」を演出で出し分け。

## 9. テスト方針（Vitest、`puzzle/` を重点）

- ピースのオリエンテーション生成数（X=1, I=2, F=8 等）が正しいか。
- 配置合法性判定（盤外・重なり）。
- 重複排除: 既知の解に 4 つの対称変換をかけた 4 盤が、すべて同一の正規形キーになること。
- 完成判定が 60 セル充足で真になること。
